1. 기술 표준 및 운영 개요

본 가이드라인은 전기차 Plug & Charge (PnC) 서비스 구현을 위해 차량 제조사 (OEM), 모빌리티 운영사업자 (eMSP), 충전사업자 (CPO) 및 V2G PKI Backend 간의 시스템 간 연계 방식과 데이터 흐름을 정의합니다. 또한, PnC운영 환경에서의 주요 시나리오를 기반으로 전기차, 충전기, CSMS (CPO) 및 PKI Backend 간 통신 인터페이스 및 메시지 규격을 명세하며, PnC구조에 적용되는 보안 체계 요구사항에 대해 기술합니다.

1.3.1 eMSP의 책임 및 시스템의 역할

  • eMSP의 책임: eMSP는 가입자의 계약 생애주기를 관리할 책임이 있습니다. 연동 방식에 따라 다음 두 가지 모델 중 하나를 선택할 수 있습니다.
    1. 자체 관리 모델: 자체 PKI 및 연산 능력을 보유한 eMSP는 직접 계약 번들을 구성하여 CPS 서명 API를 호출합니다.
    2. 위임 관리 모델: 연산 능력이 없는 eMSP는 eMSP Service API를 통해 인증서 발급부터 등록까지의 전 과정을 시스템에 위임할 수 있습니다.
  • 시스템(CPS/eMSP Service)의 역할: 시스템은 국가 V2G 신뢰 체계의 서명을 수행하고, 위임 모델의 경우 eMSP를 대신하여 보안 연산 및 데이터 정합성을 보장합니다. 서명과 저장은 하나의 논리적 단위로 처리됩니다.

1.4 인증서 지문 (Certificate Fingerprint) 규격

본 시스템에서 사용하는 모든 fingerprint 필드는 인증서의 무결성 검증 및 신뢰 앵커(Trust Anchor) 식별을 위한 고유 키로 활용됩니다.

  • 연산 알고리즘: SHA-256
  • 입력 데이터: Base64 인코딩이나 PEM 헤더가 제거된 바이너리 DER(Distinguished Encoding Rules) 형식의 인증서 원본 데이터
  • 출력 형식: 64자리 16진 문자열 (대소문자 구분 없음)
  • 주요 용도:
    1. 무결성 검증: 수신한 인증서의 해시값을 직접 계산하여 서버가 제공한 지문과 대조함으로써 전송 중 데이터 변조 여부 확인.
    2. 신뢰 매핑: 특정 계약 데이터가 어떤 Root CA를 최상위 앵커로 하는 CPS에 의해 서명되었는지 식별 (예: cpsV2GRootCertificateFingerprint).
    3. 고유 식별: UUID와 별개로 인증서 자체의 물리적 특성에 기반한 전역 고유 식별자로 활용.

2.1 PnC 전체 라이프사이클

차량 생산부터 최종 충전 계약 설치까지의 통합 흐름입니다.

sequenceDiagram
    participant OEM as OEM (차량 제조사)
    participant PCP as PCP (인증서 저장소)
    participant EV as EV (전기차)
    participant eMSP as eMSP (서비스 사업자)
    participant CPS as CPS (서명 서비스)
    participant CCP as CCP (계약 저장소)
    participant CPO as CPO (충전 사업자)

    Note over OEM, PCP: 1. 차량 생산 및 준비
    OEM->>PCP: Provisioning 인증서 등록 (PCID 기반)
    OEM->>EV: Provisioning 인증서 및 Root Trust Store 설치

    Note over eMSP, EV: 2. 서비스 가입 및 차량 할당
    eMSP->>PCP: 차량 존재 확인 및 인증서 조회 (PCID)
    PCP-->>eMSP: Provisioning 인증서 체인 반환
    eMSP->>eMSP: eMAID 생성 및 계약 번들 구성

    Note over eMSP, CCP: 3. 계약 데이터 서명 및 배포
    eMSP->>CPS: 계약 번들 서명 요청 (POST /v1/cps/contract-data/sign)
    CPS->>CCP: 서명된 계약 데이터(EXI) 자동 저장
    CPS-->>eMSP: 201 Created

    Note over EV, CCP: 4. 계약 설치 (충전 시)
    EV->>CPO: CertificateInstallationReq (EXI)
    CPO->>CCP: 계약 조회 및 EXI 구성 (POST /v1/ccp/signed-contract-data)
    CCP-->>CPO: CertificateInstallationRes (EXI)
    CPO-->>EV: CertificateInstallationRes (EXI)

sequenceDiagram
    participant eMSP as eMSP (서비스 사업자)
    participant CPS as CPS (서명 서비스)

4.1 인증 기술 옵션 및 검토 배경

상호운용성과 보안 강도 사이의 균형을 위해 다음 두 가지 방식을 기술 후보로 고려하고 있습니다. 최종 방식은 보안성 검토 및 사업자 협의를 거쳐 확정될 예정입니다.

  • 옵션 1: mTLS (Mutual TLS)
    • 특징: 하드웨어(HSM) 기반 인증서 관리가 가능한 사업자 환경에 적합하며, 인프라 수준에서 가장 강력한 S2S 인증을 제공합니다.
  • 옵션 2: OAuth2 (JWT/Client Credentials)
    • 특징: 토큰 기반의 유연한 권한 관리. 세부 권한(Scope) 제어가 용이하며, 클라우드 네이티브 환경에 최적화됨.

4.2 인증 식별자와 헤더의 연계 검증 (Identity Linkage)

인증 방식과 관계없이, 호출자의 사칭(Impersonation)을 방지하기 위해 인증 계층의 주체 정보와 애플리케이션 헤더(KPNC-From-Party-Id) 간의 연계 검증이 수행되어야 합니다.

  • 검증 메커니즘 (안):
    1. Identity Extraction: 인증 과정에서 클라이언트의 고유 식별자(인증서 CN 또는 JWT Sub)를 추출합니다.
    2. Header Comparison: 추출된 식별자가 API 요청 헤더인 KPNC-From-Party-Id와 논리적으로 일치하는지 대조합니다.
    3. Policy Enforcement: 두 식별 정보가 불일치할 경우, 인증되지 않은 사용자의 사칭 시도로 간주하여 401 Unauthorized를 반환합니다.

4.3 역할 및 권한 (RBAC)

인증된 주체의 역할(Party Type)에 따라 리소스별 접근 권한이 차등 부여됩니다.

  • OEM: PCID의 WMI(첫 3자리) 기반 소유권 검증 및 자사 차량 데이터 관리 권한.
  • eMSP: eMAID 접두사 기반의 가입자 계약 생애주기 관리 및 CPS 서명 요청 권한.
  • CPO: 실시간 충전 프로토콜 수행을 위한 CCP 계약 데이터 조회 권한.
  • Trust Operator: 본 시스템의 운영 주체로, Root 인증서 등록/삭제 및 시스템 전반의 관리 권한 보유.

5. 속도 제한 정책

시스템의 안정적인 운영과 공정한 자원 배분을 위해 모든 API 엔드포인트에 대해 속도 제한이 적용됩니다.

  • 제한 방식: 파티(Party)별, API 도메인별 단위 시간당 요청 횟수 제한
  • 응답 헤더: 모든 API 응답(성공/실패 포함)에는 다음 헤더가 포함될 수 있습니다.
    • KPNC-Rate-Limit-Limit: 허용된 최대 요청 수
    • KPNC-Rate-Limit-Remaining: 현재 남은 요청 가능 수
    • KPNC-Rate-Limit-Reset: 초기화까지 남은 시간 (초)
  • 초과 시 대응: 제한 초과 시 429 Too Many Requests 에러가 반환되며, Retry-After 헤더를 통해 재시도 가능 시간을 안내합니다.

[Note]

본 명세서의 가독성을 위해 성공 응답(2xx)에는 위 헤더들을 명시적으로 정의하지 않았으나, 실제 런타임에서는 모든 응답에 포함되어 전달됩니다. 429 Too Many Requests 응답 시에는 클라이언트의 즉각적인 대응(Retry-After 확인 등)을 돕기 위해 헤더 정보가 명세에 상세히 기술되어 있습니다.

연산 능력이 없는 eMSP를 위한 비즈니스 오케스트레이션. 인증서 발급부터 CCP 등록까지의 전 과정을 대행합니다.

계약 발급 및 등록 대행

연산 능력이 부족한 eMSP를 위한 비즈니스 오케스트레이션 API입니다. 호출 시 시스템은 내부적으로 다음 절차를 단일 트랜잭션으로 수행합니다:

  1. 차량 검증: PCP를 통해 대상 PCID의 유효성 및 Provisioning 인증서 확인
  2. 키 생성: 보안 영역(HSM 등)에서 계약용 키 쌍 생성
  3. 인증서 발급: PKI 서비스를 통해 eMSP Root/Sub-CA로부터 계약 인증서 발급
  4. 데이터 암호화: 차량의 Provisioning Public Key를 사용하여 프라이빗 키 암호화
  5. CPS 서명: CPS를 통한 V2G 기술 서명 추가 및 최종 EXI 데이터 완성
  6. 배포 저장: 완성된 데이터를 CCP에 즉시 등록

이 과정을 통해 eMSP는 복잡한 암호 연산 없이도 PnC 계약 발급 비즈니스를 완결할 수 있습니다. 응답으로 반환되는 status는 RFC 6960(OCSP) 규격을 따릅니다.

PCP (Provisioning Certificate Pool)

OEM 차량 Provisioning 인증서 저장 및 PCID 조회를 담당합니다.

Provisioning 인증서 등록/갱신

차량의 고유 식별자(PCID)를 경로 식별자로 사용하여 Provisioning 인증서 체인을 PCP에 등록하거나 갱신합니다 (Upsert).

본 API는 주로 차량 제조사(OEM) 백엔드에서 호출하며, 차량이 공장에서 출고되기 전 혹은 현장에서 Provisioning 인증서가 갱신될 때 사용됩니다. 등록된 인증서는 PnC 충전 시 eMSP가 차량의 유효성을 검증하고 계약 인증서를 암호화하여 전달하기 위한 신뢰의 기초가 됩니다.

[WMI 기반 검증] 시스템은 요청된 PCID의 첫 3자리인 WMI(World Manufacturer Identifier)를 추출하여 해당 요청자가 해당 제조사의 차량 데이터를 관리할 권한이 있는지 엄격히 검증합니다. 이는 시스템의 무결성을 유지하는 가장 기초적이고 중요한 보안 메커니즘입니다.

[서버 검증 절차 예시]

  1. WMI 권한 검증: PCID의 WMI가 해당 OEM 계정에 할당된 권한과 일치하는지 확인합니다.
  2. 유효 기간 검증: 리프 인증서부터 Root까지 모든 체인의 유효 기간이 현재 시각 이후인지 확인합니다.
  3. 상태 검증: 각 인증서의 검증 서비스(OCSP)를 통해 폐지 여부를 실시간 확인합니다.
  4. 신뢰 체인 검증: 등록된 국내 PnC Root 인증서로부터 리프까지의 신뢰 체인(Trust Chain)을 엄격히 검증합니다.

ISO 15118 규격에 따라 PCID와 인증서 체인 간의 무결성을 엄격히 검증하며, 동일한 PCID로 요청 시 최신 정보로 교체됩니다.

CPS (Contract Provisioning Service)

계약 데이터에 V2G 신뢰 체계 기반의 CPS 서명을 수행하고 CCP의 데이터를 총괄 관리하여 배포 준비를 수행합니다.

계약 데이터 서명

eMSP가 자체 PKI를 통해 발급한 계약 인증서 번들을 V2G 신뢰 체계(V2G PKI)로 서명하여 완성된 계약 데이터를 생성합니다. 이 과정은 ISO 15118 PnC 표준에서 요구하는 이중 서명 구조(eMSP 서명 + V2G 서명)를 완성하는 핵심 단계입니다.

[역할 및 책임 분담]

  • eMSP: 계약 인증서 체인 발급, 암호화된 프라이빗 키 생성, DH 공개키 추출 등을 포함한 전체 계약 번들 구성을 완료하여 제출해야 합니다. (eMSP의 자체 PKI 보유 필수)
  • CPS: 제출된 번들을 검증한 후, V2G Root 계층의 서명을 추가하여 최종 계약 데이터(EXI 기반)를 완성하고 이를 CCP(Contract Certificate Pool)에 원자적으로 저장하는 역할을 수행합니다.

본 API의 성공 응답(201 Created)은 V2G 서명 완료와 CCP 내 물리적 저장 및 배포 준비가 하나의 트랜잭션으로 완료됨을 보장합니다. 따라서 호출자는 별도의 정합성 검증이나 추가 저장 요청 없이 즉시 CCP를 통해 계약 조회가 가능합니다. 국가 PnC 보안 가이드라인에 따라 모든 데이터는 보안이 유지되는 채널을 통해 전송되어야 합니다.

서명된 계약 데이터 조회 (OEM용)

특정 eMAID와 PCID의 조합으로 서명된 계약 데이터를 조회합니다. 주로 OEM 백엔드가 자사 차량(PCID)에 할당된 특정 사용자의 계약을 서버 간 통신(S2S)을 통해 가져올 때 사용합니다.

복합 식별자({emaid}/{pcid}) 사용 사유: ISO 15118 표준에서 '서명된 계약 데이터(EXI)'는 단순히 계약(eMAID)의 상태 정보가 아니라, 해당 계약을 설치할 특정 차량(PCID)의 Provisioning 인증서로 암호화되어 생성된 물리적 데이터 덩어리입니다. 동일한 계약(eMAID)이라 하더라도 대상 차량(PCID)이 다르면 데이터의 내용(EXI 스트림) 자체가 완전히 달라지므로, 차량과 계약의 쌍을 고유 리소스로 취급하여 관리합니다.

응답 형식에 관한 참고: 본 API는 ISO 15118-20 규격의 '다중 계약 설치/갱신' 시나리오에 대비하여 단건 조회임에도 컬렉션(배열) 형식을 반환합니다. ISO 15118-2 환경에서는 일반적으로 조건에 부합하는 1건의 데이터만 배열에 포함되어 반환됩니다.

요청 시 'sessionId' 파라미터를 제공하면, 반환되는 EXI 응답 내에 해당 세션 ID가 포함되어 차량과의 보안 세션 연결을 보장할 수 있습니다. 이 API는 차량 제조사가 사용자의 요청에 따라 계약 데이터를 사전 다운로드(Push model)할 때 핵심적인 역할을 수행합니다. 보안을 위해 eMAID와 PCID의 소유권 및 매핑 관계를 엄격히 검증합니다.

2. 전달 보장 정책 (Delivery Policy)

시스템은 일시적인 네트워크 장애나 수신 측 서버의 단기 순단에 대응하기 위해 다음과 같은 공격적인 재시도 정책을 시행합니다.

  • 재시도 전략: 지수 백오프(Exponential Backoff) 기반 자동 재시도.
  • 재시도 간격 및 횟수: 총 5회 시도 (간격: 1초, 4초, 16초, 64초, 256초).
  • 최대 가용 시간: 최초 발생 시점으로부터 약 6분(341초) 동안 재시도를 수행합니다.
  • 실패 처리: 5회 재시도 후에도 성공(2xx) 응답을 받지 못할 경우, 해당 이벤트 전달은 최종 실패 처리되며 시스템 대기열에서 제거됩니다.
  • 상태 코드 대응: 타임아웃(10초 이내 무응답) 또는 2xx 이외의 모든 응답(4xx, 5xx) 발생 시 재시도를 트리거합니다.

3. 데이터 정합성 보장을 위한 연동사 준수 사항

Webhook은 비동기 알림 수단이며, 국가 시스템의 재시도 정책은 '단기 장애 대응'에 초점이 맞춰져 있습니다. 연동 업체는 시스템 간 데이터 무결성을 보장하기 위해 다음 사항을 반드시 구현해야 합니다.

  • 주기적 정합성 대조 (Polling): 연동사의 시스템 장애가 재시도 정책 시간(6분)보다 길어질 경우를 대비하여, 각 서비스 도메인(CCP, PCP 등)의 조회 API를 활용한 주기적 데이터 동기화 로직을 반드시 갖추어야 합니다.
  • 멱등성(Idempotency) 확보: 재시도 간격이 짧으므로 동일한 이벤트가 중복 전달될 가능성이 높습니다. 수신 측에서는 eventId를 식별자로 사용하여 중복 처리 방지 로직을 반드시 구현해야 합니다.
  • 실시간 공정(EOL) 대응: 차량 제조 라인(EOL) 등 즉각적인 피드백과 차단형(Blocking) 처리가 필요한 중요한 로직에는 웹훅에만 의존하지 말고, API 응답값 또는 즉시 조회 API를 병행 사용하십시오.

Webhook 등록

새로운 Webhook 엔드포인트를 등록합니다.

연동 개발자 가이드:

  1. URL: 이벤트를 수신할 서버의 HTTPS 공개 URL이어야 합니다.
  2. Enabled Events: 수신하고자 하는 이벤트 코드를 지정합니다 (예: contract.created.for.oem).
  3. WMI/ProviderId: 특정 제조사나 사업자의 이벤트만 필터링하려면 지정합니다.

보안 및 검증 (Security & Verification): 시스템은 Webhook 메시지의 무결성을 보장하기 위해 모든 요청 헤더에 KPNC-Signature를 포함합니다. 수신 측에서는 다음 절차에 따라 요청을 검증해야 합니다.

  1. 방식: HMAC-SHA256
  2. 키 (Key): 엔드포인트 등록 시 발급된 secret
  3. 데이터 (Data): HTTP Request Body (JSON String 전체)
  4. 검증: 생성된 해시값과 KPNC-Signature 헤더값이 일치하는지 확인 (Constant-time comparison 권장)

코드 예시 (Java/Spring):

@PostMapping("/webhooks")
public ResponseEntity<Void> handleWebhook(@RequestBody String payload, 
                                         @RequestHeader("KPNC-Signature") String signature) {
    String expected = hmacSha256(payload, secret);
    if (!MessageDigest.isEqual(expected.getBytes(), signature.getBytes())) {
        return ResponseEntity.status(HttpStatus.UNAUTHORIZED).build();
    }
    // 이벤트 처리 로직
    return ResponseEntity.ok().build();
}

등록 성공 시 반환되는 secret 값은 이후 Webhook 메시지의 무결성 검증(HMAC SHA-256)에 사용되므로 반드시 안전하게 보관해야 합니다.
